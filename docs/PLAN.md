# cliproxy-activator 对齐 yunyi-activator 的差异分析与实施计划

更新时间：2026-04-05

本文档基于以下对象整理：

- 本地仓库：`/opt/cliproxy-activator`
- 对标包：`yunyi-activator@1.8.2`
- 对标方式：读取本地实现，解包并静态分析 npm 发布包 `yunyi-activator-1.8.2.tgz`

## 1. 结论摘要

当前 `cliproxy-activator` 还处于 CLI 脚手架阶段，和真实 `yunyi-activator` 的差距不是“补几个插件”这么简单，而是整体上还缺少完整的激活闭环。

如果只按“插件”狭义统计：

- 还缺少 2 个编辑器插件集成：
  - VSCode Codex 插件
  - Cursor Codex 插件

如果按“真实包会顺带配置的客户端/插件能力”统计：

- 还缺少 4 个集成项：
  - VSCode Codex 插件
  - Cursor Codex 插件
  - OpenCode
  - OpenClaw

但更大的差距在于：

- 当前仓库只有命令入口和交互菜单
- `activate`、`test`、`restore` 仍是占位实现
- 真实包已经包含 API Key 校验、节点测速、备份恢复、配置写入、编辑器插件补丁等完整流程

## 2. 当前仓库现状

### 2.1 已有能力

- 命令入口已存在
- 交互菜单已存在
- 暴露了 `activate`、`test`、`restore`、`interactive`
- 可以选择服务：
  - Claude Code
  - Codex

  - OAuth

### 2.2 当前明确未完成的地方

- 后端激活请求未接入
- 节点发现、探测、评分、回退未接入
- 备份存储与恢复逻辑未接入
- 本地配置文件写入未接入
- 插件与客户端集成未接入
- 前端资源只是备份资产，未和 CLI 串联

### 2.3 现状判断

当前项目更像：

- CLI 壳子
- 未来功能入口草图

而不是：

- 可直接替代 `npx yunyi-activator` 的生产工具

## 3. 对标包真实能力

基于 `yunyi-activator@1.8.2` 的发布包静态分析，可以确认它至少具备以下能力。

### 3.1 核心服务能力

- 支持两类服务：
  - Claude Code
  - Codex
- 使用 API Key 激活流
- 校验 API Key 状态
- 根据服务类型写入不同配置

### 3.2 节点能力

- 节点列表探测
- 多节点测速
- 平均延迟与稳定性评分
- 推荐最优节点
- 常规节点不可用时尝试备用节点
- 在网络波动时提示用户是否切到备用节点

### 3.3 备份与恢复

- 激活前创建备份
- 分类备份 Claude / Codex
- 展示备份列表
- 恢复指定备份
- 清理全部备份

### 3.4 本地配置写入

- Claude 相关配置写入
- Codex 相关配置写入
- 写入 API Key、模型别名、认证模式等配置
- 配置完成后提示重启相关工具

### 3.5 附加客户端集成

- OpenCode
- OpenClaw

这两个不是“编辑器插件”，更准确地说是额外客户端配置集成。

### 3.6 编辑器插件集成

真实包会检测并处理以下 Codex 编辑器扩展：

- VSCode
- Cursor

并且会：

- 检查扩展安装目录
- 找到扩展资源文件
- 对资源文件打补丁，使扩展处于目标激活状态

### 4.3 产品设计差异

这是最需要先定下来的地方：

- 本地项目保留了 `OAuth` 选项
- 真实 `yunyi-activator` 并没有 `OAuth`
- 真实包的核心闭环是 `API Key -> 校验 -> 节点选择 -> 写配置 -> 备份/恢复`

所以如果目标是“先对齐真实包”，那么当前 `OAuth` 不应作为第一阶段主线。

建议：

- 第一阶段先实现 API Key 闭环
- `OAuth` 作为第二阶段扩展能力
- 在 `OAuth` 未落地前，不应对用户显示为“可用完成态”

## 5. 目标定义

建议把目标拆成两层。

### 5.1 目标 A：先达到对齐 yunyi-activator 的最小可替代能力

完成后，应满足：

- 用户能通过 CLI 输入 API Key
- 工具能验证 Key 状态
- 工具能探测节点并选择节点
- 工具能写入 Claude / Codex 本地配置
- 工具能创建备份
- 工具能恢复备份
- 工具能自动处理 OpenCode / OpenClaw
- 工具能检测并处理 VSCode / Cursor Codex 插件

### 5.2 目标 B：再做本项目自己的扩展能力

扩展项包括：

- OAuth 登录流
- 与 CLIProxyAPI 后端的专属联动
- 复用现有 `management-center` / `code-center` 资源
- 更完整的图形化管理中心

## 6. 建议范围控制

为避免项目失控，本轮实施建议明确以下边界。

### 6.1 本轮必须完成

- API Key 激活闭环
- 节点测试
- 备份与恢复
- Claude / Codex 配置写入
- OpenCode / OpenClaw 集成
- VSCode / Cursor 插件处理

## 7. 建议技术方案

当前仓库所有逻辑几乎都在 `lib/index.js`，不适合继续堆功能。建议先重构模块边界，再补功能。

### 7.1 目标目录结构

建议拆成如下结构：

```text
bin/
  cli.js
lib/
  index.js
  cli/
    menu.js
    prompts.js
    output.js
  commands/
    activate.js
    test.js
    restore.js
  services/
    activate-service.js
    config-service.js
    backup-service.js
    node-service.js
    verify-service.js
  providers/

  config/

    codex.js
    opencode.js

  integrations/
    vscode-codex.js

  backup/
    index.js
    store.js
  utils/
    fs.js
    path.js
    json.js
    env.js
    errors.js
```

### 7.2 核心模块职责

`commands/activate.js`

- 驱动完整激活流程
- 负责调用节点测试、Key 校验、配置写入、插件处理、备份创建

`services/verify-service.js`

- 校验 API Key
- 解析返回状态
- 标准化状态：
  - `active`
  - `inactive`
  - `disabled`
  - `expired`
  - `unknown`

`services/node-service.js`

- 节点列表管理
- 延迟测试
- 稳定性评分
- 备用节点切换

`services/backup-service.js`

- 创建备份
- 列表查询
- 恢复
- 清理

`config/*.js`

- 面向不同客户端生成配置内容
- 封装读写路径
- 封装原子写入

`integrations/*.js`

- 查找编辑器扩展目录
- 检查目标文件
- 打补丁
- 回写备份元信息

## 12. 风险与未知项

### 12.1 最大风险：后端协议未冻结

如果 CLIProxyAPI 的接口结构和真实 `yunyi-activator` 所依赖接口不同，会影响：

- Key 校验
- 服务类型识别
- 节点列表获取
- 激活状态回显

应对：

- 先定义 provider 适配层
- 不把协议细节散落到命令代码里

### 12.2 第二风险：插件补丁脆弱

VSCode / Cursor 扩展资源文件一旦升级，字符串匹配规则可能失效。

应对：

- 补丁逻辑必须版本感知
- 补丁失败要可回退
- 补丁规则要有 fixture 测试

### 12.3 第三风险：本地路径兼容性

不同系统和不同安装方式可能导致：

- 配置路径不同
- 扩展路径不同
- 权限行为不同

应对：

- 所有路径检测集中管理
- 用探测逻辑代替硬编码

### 12.4 第四风险：OAuth 过早介入

如果第一阶段同时做 API Key 和 OAuth，极易拖慢交付。

应对：

- 本阶段冻结为 API Key 主线
- OAuth 只保留接口，不保交付

## 13. 推荐实施顺序

强烈建议按下面顺序做，不要倒序：

1. 先拆模块
2. 再做 API Key 校验
3. 再做节点测试
4. 再做配置写入和备份
5. 再做 OpenCode / OpenClaw
6. 再做 VSCode / Cursor 插件
7. 最后补测试、README、发包

不建议的顺序：

- 先做前端
- 先做 OAuth
- 先做插件补丁

原因：

- 都不是闭环最短路径
- 也都依赖前面的基础层稳定
